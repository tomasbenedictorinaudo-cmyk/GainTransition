1. Problem Statement

A satellite communications payload consists of X receive antennas and Y transmit antennas connected through a multi-stage gain chain. Each logical channel traverses a fixed path of 7 gain stages from receive to transmit. The operator must reconfigure gain settings across all stages from a current operating point to a new target configuration.

The transition cannot be applied atomically—each gain stage is adjusted individually in discrete increments determined by its hardware granularity. During the transition, the payload continues to serve active traffic. The problem is to find a sequence of gain adjustments that:

  1. Minimizes transient EIRP disruption per channel. In greedy and inner-first modes, the algorithm maximizes gain progress (finishing the transition as fast as possible). In G4-compensated mode, it actively steers EIRP back toward initial values between primary steps.
  2. Never violates power thresholds at any gain stage output during any intermediate state. Each stage has hardware-defined upper and lower power limits.
  3. Respects EIRP deviation limits when configured. A hard constraint rejects any move that would cause any channel's EIRP to deviate beyond the allowed negative or positive envelope from its initial value.
  4. Respects discrete granularity constraints: each gain stage can only be adjusted in multiples of its specific step size.
  5. Terminates with all gain stages at their target values.
Operational context. In geostationary satellite payloads, gain reconfigurations occur during beam repointing, traffic rebalancing, or interference mitigation. A naive sequential approach—adjusting one stage at a time—can cause transient EIRP excursions of tens of dB, triggering regulatory violations or link margin exhaustion. CCGS reduces these transients to the minimum physically achievable given the discrete step constraints.

2.3 Coupling

The coupling map records, for each gain stage instance, which channels flow through it. This is built from the channel definitions:

  • G1:rxAnt{a} → all channels with rxAntennaId = a
  • G2:rxAnt{a}:sub{s} → all channels whose Rx frequency band overlaps subchannel s on antenna a
  • G3:ch{id}, G4:ch{id}, G5:ch{id} → channel id only
  • G6:txAnt{b}:sub{s} → all channels whose Tx frequency band overlaps subchannel s on antenna b
  • G7:txAnt{b} → all channels with txAntennaId = b

2.4 Gain Stage Key Format

Each gain stage instance is identified by a string key of the form TYPE:scope. Examples:

  • G1:rxAnt0 — LNA for receive antenna 0
  • G2:rxAnt0:sub2 — DEMUX for subchannel 2 on receive antenna 0
  • G4:chA — per-channel G4 for channel A
  • G7:txAnt1 — HPA for transmit antenna 1

3. Mathematical Formulation

3.1 State Space

Definition: System State. A state S is a vector of all gain values: S = { gk ∈ ℝ : k ∈ K } where K is the set of all gain stage keys and each gk is expressed in dB. The initial state is S0 and the target state is ST.

Each gain stage k has:

  • Current value gk(0) (dB)
  • Target value gk(T) (dB)
  • Granularity γk (dB) — the minimum discrete step size
  • Remaining steps rk = round(|gk(T) − gk| / γk)
  • Direction dk = sign(gk(T) − gk) ∈ {−1, +1}

Convergence is reached when for all k ∈ K:

|gk − gk(T)| < 0.01 · γk

3.2 EIRP Computation

For channel c with gain chain [k1, k2, …, k7], the EIRP in dBm is:

EIRPc = Pin,c + ∑j=1..7 gkj

where the input power is:

Pin,c = IPFDc + 10 · log10(BWc · 106)

Variable definitions:

  • IPFDc: input power flux density for channel c (dBm/m²)
  • BWc: channel bandwidth in MHz
  • gkj: gain value of the j-th stage in channel c's chain (dB)

The EIRP deviation from initial is:

ΔEIRPc = EIRPc(current) − EIRPc(initial)

3.3 Power Level Cascade

The output power at each stage is computed by cascading through the chain. For a single channel c with chain [k1, …, k7]:

Pout(kj, c) = Pin,c + ∑i=1..j gki

For per-channel stages (G3, G4, G5), the power level at the stage output equals the single channel's cascaded power.

For shared stages (G1, G2, G6, G7), the total power at the stage output is the sum in the linear domain of all channels flowing through that stage:

Ptotal(k) = 10 · log10( ∑c ∈ C(k) 10Pout(k, c) / 10 ) [dBm]

where C(k) is the set of channels coupled to gain stage k (from the coupling map).

Linear aggregation. Power levels must be summed in the linear domain (milliwatts), not in dB. The conversions are: linear(P) = 10P/10     dBm(p) = 10 · log10(p)

3.4 Constraints

3.4.1 Power Threshold Constraints

At every intermediate state, for every gain stage k:

Plower(k) ≤ Ptotal(k) ≤ Pupper(k)

where Plower(k) and Pupper(k) are the hardware-defined thresholds in dBm.

3.4.2 EIRP Deviation Constraints

When configured (non-null), for every channel c at every intermediate state:

−L− ≤ ΔEIRPc ≤ L+

where L− = maxNegativeEirpDeviation and L+ = maxPositiveEirpDeviation (both in dB). A tolerance of 0.001 dB is applied to avoid floating-point false rejections.

3.5 System Noise Temperature

The Friis cascade formula for the receive chain (G1, G2, G3) of channel c:

Tsys,c = Tant,c + T1 + T2 / G1lin + T3 / (G1lin · G2lin)

where:

  • Tant,c: antenna noise temperature for channel c (K)
  • Tn = T0 · (Fn − 1): equivalent noise temperature of stage n
  • Fn = 10NFn/10: noise factor (linear) from noise figure NFn (dB)
  • Gnlin = 10gn/10: gain in linear from gain in dB
  • T0 = 290 K: standard reference temperature

The general form for N receive stages:

Tsys,c = Tant,c + ∑n=1..N Tn / ∏m=1..n−1 Gmlin

(The product is 1 when n = 1.) Only G1, G2, G3 contribute to the receive chain; G4–G7 are transmit-side.

4. Algorithm Description

4.1 Overview

CCGS is an iterative greedy algorithm. On each iteration it:

  1. Generates all candidate moves (one per gain stage type with remaining delta, for each possible step multiplier).
  2. Filters out candidates that violate power thresholds.
  3. Filters out candidates that violate EIRP deviation limits (when configured).
  4. Scores remaining candidates by total gain progress and selects the best.
  5. Applies the move, records the new state, checks convergence.

Three strategy modes exist: greedy, inner-first, and g4-compensated. The first two use the standard loop (differing only in tiebreaking). The third uses a two-phase approach described in Section 8.

4.2 Standard Algorithm Pseudocode

function runStandard(channels, gainStages, params):
    {gainValues, targetValues, granularities} = initState(gainStages)
    initialEirp = computeAllChannelEirp(channels, gainValues)
    initialSystemTemp = computeAllSystemTemp(channels, gainValues, gainStages)
    hasEirpLimits = (params.maxNegativeEirpDeviation != null OR
                     params.maxPositiveEirpDeviation != null)

    steps = []
    maxNegDev = 0;  maxPosDev = 0;  violations = 0

    for iter = 0 to params.maxIterations - 1:
        if allAtTarget(gainValues, targetValues, granularities): break

        // Step 1: Generate candidate moves
        candidates = generateCandidateMoves(gainValues, targetValues, granularities)
        if candidates is empty: break

        // Step 2: Filter by power thresholds
        feasible = filter candidates by checkFeasibility(move, gainValues, channels, gainStages)

        if feasible is empty:
            // Deadlock: relax thresholds by +/- 1 dB
            feasible = filterRelaxedPower(candidates, gainValues, channels, gainStages)
            if feasible is empty: break   // permanently stuck
            violations += 1

        // Step 3: Filter by EIRP limits (hard constraint)
        if hasEirpLimits:
            eirpFiltered = filterByEirpLimits(feasible, gainValues, channels, initialEirp, params)
            if eirpFiltered is empty:
                return ERROR with best achievable deviation
            feasible = eirpFiltered

        // Step 4: Score and select
        best = pickBest(feasible, gainValues, channels, initialEirp, params)

        // Step 5: Apply and record
        applyMove(best, gainValues)
        step = recordStep(steps.length, best, gainValues, channels, gainStages, initialEirp)
        steps.push(step)
        trackDeviation(step, maxNegDev, maxPosDev)

    return buildResult(steps, ...)

4.3 Detailed Iteration Flowchart

4.4 Initialization Procedure

Before the first iteration, the following setup occurs:

  1. Extract state vectors. For each gain stage entry in the gainStages map, copy currentValue → gainValues[key], targetValue → targetValues[key], stepGranularity → granularities[key].
  2. Compute initial EIRP. For each channel, evaluate EIRPc = Pin,c + ∑ gk over the channel's gain chain. Store as initialEirp.
  3. Compute initial system temperature. For each channel, evaluate Tsys,c using the Friis cascade formula. Store as initialSystemTemp.
  4. Build coupling map (for G4-compensated strategy only). Map each gain stage key to the list of channel IDs it affects.
  5. Determine EIRP limit mode. Set hasEirpLimits = true if either maxNegativeEirpDeviation or maxPositiveEirpDeviation is non-null.

5. Candidate Generation

5.1 Rules

At each iteration, candidate moves are generated for every gain stage type that has at least one instance with remaining delta. The key constraint is: each move changes only one gain stage type.

5.1.1 Analog Stages (G1, G7)

For each individual antenna instance with remaining delta, generate one candidate per step multiplier:

for each pending antenna instance (key, maxSteps, gran, direction):
    for n = 1 to maxSteps:
        emit CandidateMove {
            steps: [{ gainStageKey: key, delta: direction * n * gran }],
            stageType: type
        }

This means a G1 with 5 remaining steps produces 5 candidates (1-step, 2-step, ..., 5-step), each affecting only that one antenna.

5.1.2 Digital Stages (G2–G6)

All instances of the same type move together in a single candidate. The step multiplier ranges from 1 to the maximum remaining steps across all instances. Each instance is individually capped at its own remaining steps:

maxMultiplier = max(remainingSteps across all instances of this type)

for n = 1 to maxMultiplier:
    steps = []
    for each pending instance (key, maxSteps, gran, direction):
        actualN = min(n, maxSteps)
        steps.push({ gainStageKey: key, delta: direction * actualN * gran })
    emit CandidateMove { steps, stageType: type }
Capping semantics. When the multiplier n exceeds an instance's remaining steps, that instance contributes its maximum delta (reaching its target). Other instances in the same move may still have room to step further. This produces asymmetric moves where some instances finish early.

5.2 Example

Consider G3 with three channels: ch A (3 steps remaining, +0.5 dB gran), ch B (1 step remaining, +0.5 dB gran), ch C (2 steps remaining, −0.5 dB gran).

Multiplier nch A deltach B deltach C delta

5.3 Excluding Stage Types

The candidate generator accepts an optional excludeTypes set. In G4-compensated Phase 1, excludeTypes = {'G4'} is passed to prevent G4 from appearing as a primary move candidate (G4 is instead used for compensation steps).

5.4 Complexity of Candidate Generation

Let S be the number of gain stage types with remaining delta, A the number of analog antenna instances, D the number of digital instances, and M the maximum remaining steps for any instance. Then the number of candidates per iteration is bounded by:

|candidates| ≤ A · M + Sdigital · M

where Sdigital is the number of digital stage types with pending delta. In typical payloads (tens of channels, M ≤ 20), this produces hundreds of candidates per iteration.

6. Feasibility Verification

6.1 Power Threshold Check

For each candidate move, the algorithm tentatively applies the move to a copy of the current state, then checks all power levels:

function checkFeasibility(move, currentGainValues, channels, gainStages):
    tempGains = copy(currentGainValues)
    for each step in move.steps:
        tempGains[step.gainStageKey] += step.delta

    powerLevels = computeAllPowerLevels(channels, tempGains)

    for each (key, power) in powerLevels:
        stage = gainStages.get(key)
        if power > stage.upperThreshold + 0.001: return false
        if power < stage.lowerThreshold - 0.001: return false

    return true

The 0.001 dB tolerance prevents floating-point rounding from causing false rejections.

6.2 EIRP Limit Check (Standard Mode)

When EIRP limits are configured, each power-feasible candidate is additionally checked against the EIRP envelope:

function filterByEirpLimits(candidates, gainValues, channels, initialEirp, params):
    negLimit = params.maxNegativeEirpDeviation
    posLimit = params.maxPositiveEirpDeviation

    return candidates.filter(move =>
        tempGains = copy(gainValues)
        for each step in move.steps:
            tempGains[step.gainStageKey] += step.delta

        eirp = computeAllChannelEirp(channels, tempGains)

        for each channel c:
            dev = eirp[c.id] - initialEirp[c.id]
            if negLimit != null AND dev < -(negLimit + 0.001): reject
            if posLimit != null AND dev >  (posLimit + 0.001): reject

        accept
    )

6.3 EIRP Limit Check (G4-Compensated Mode)

In G4-compensated mode, the EIRP check is more involved because it must verify all intermediate states in the two-step sequence. For "after" mode:

  1. State 1 (post-primary, pre-G4): Apply the primary move. Compute EIRP. Check limits.
  2. State 2 (post-G4 correction): Compute the G4 correction from the deviation. Apply it. Compute EIRP. Check limits.

Both states must satisfy the EIRP constraints. For "before" mode, the order is reversed: G4 pre-correction first, then primary step.

6.4 Infeasible EIRP Limits

When no candidate survives the EIRP filter, the algorithm computes the best achievable deviation among all power-feasible candidates and returns an error:

for each power-feasible candidate:
    apply move temporarily
    compute EIRP for all channels
    worstDev = max |EIRP_c - initialEIRP_c| over all channels
    track best (smallest) worstDev

return error: "EIRP limits infeasible. Smallest achievable worst-case: +/-{bestWorstDev} dB"

7. Scoring & Selection

7.1 Progress-Based Scoring

Every candidate that passes the feasibility filters is scored by total gain progress:

score(move) = − ∑i |δi|

where δi is the gain delta of the i-th atomic step in the move. Lower score = better (more progress). This means a move that changes 3 channels by 0.5 dB each has score −1.5, while a move changing 1 channel by 0.5 dB has score −0.5.

Why pure progress? EIRP deviation limits are enforced as hard filters upstream. Any candidate reaching the scoring stage has already been verified to stay within the allowed EIRP envelope. Scoring purely by progress lets the algorithm exploit the full allowed headroom to finish the transition in fewer steps.

7.2 Selection with Tiebreaking

Candidates are sorted by score (ascending). When two candidates have the same score (within 0.001 tolerance), tiebreaking rules apply:

function pickBest(feasible, gainValues, channels, initialEirp, params):
    scored = feasible.map(move => { move, score: scoreCandidate(move, ...) })
    scored.sort((a, b) =>
        diff = a.score - b.score
        if |diff| > 0.001: return diff           // primary: more progress
        return compareCandidates(a.move, b.move, params)  // tiebreak
    )
    return scored[0].move

Definition: Inner stages. The set {G3, G4, G5} are "inner" (per-channel) stages. The inner-first strategy prefers adjusting these before shared (outer) stages, which typically produces smaller EIRP transients since per-channel adjustments don't affect other channels.

8. G4-Compensated Strategy

The G4-compensated strategy uses G4 (a per-channel gain stage) as a balancing knob to counteract EIRP deviations caused by other stage changes. It operates in two phases.

8.1 Phase 1: Non-G4 Gains with G4 Correction

Phase 1 iterates until all non-G4 gains reach their targets. On each iteration:

  1. Generate candidate moves excluding G4 (by passing excludeTypes = {'G4'}).
  2. Filter candidates by power thresholds (with deadlock relaxation if needed).
  3. Filter candidates by EIRP limits, checking all intermediate states including the G4 correction (see Section 6.3).
  4. Score and select the best primary move.
  5. Apply the primary move and its G4 correction in the configured order ("after" or "before" mode).
  6. Record both the primary step and the G4 correction step as separate entries in the transition plan.

8.1.1 "After" Mode (Recommended)

The default mode. Sequence per iteration:

  1. Apply the primary move (e.g., change all G3 instances).
  2. Record the resulting state as a transition step.
  3. Compute G4 correction: for each channel, compute EIRP deviation from initial, quantize the inverse to G4 granularity, produce G4 adjustment steps.
  4. Apply the G4 correction (all affected G4 instances change simultaneously).
  5. Record the post-correction state as another transition step.

8.1.2 "Before" Mode

The G4 pre-correction is computed by predicting what the primary move will do, then applying the inverse G4 adjustment before the primary move:

  1. Simulate the primary move on a copy of the state to determine post-primary EIRP.
  2. Compute G4 correction from the predicted deviation.
  3. Apply the G4 pre-correction. Record the state.
  4. Apply the primary move. Record the state.

8.2 G4 Correction Computation

The G4 correction is computed per-channel as follows:

function buildG4StepsFromDeviation(referenceEirp, actualEirp, channels, currentGainValues, granularities, gainStages):
    g4Steps = []

    for each channel c:
        deviation = actualEirp[c.id] - referenceEirp[c.id]       // dB
        if |deviation| < 0.001: continue

        g4Key = "G4:ch" + c.id
        g4Gran = granularities[g4Key]

        rawCorrection = -deviation                                // invert the deviation
        numSteps = round(rawCorrection / g4Gran)                  // quantize to granularity
        if numSteps == 0: continue

        g4Steps.push({ gainStageKey: g4Key, delta: numSteps * g4Gran })

    if g4Steps is empty: return null
    return CandidateMove { steps: g4Steps, stageType: 'G4' }

Key formula for the quantized G4 correction on channel c:

ΔG4c = round(−ΔEIRPc / γG4,c) · γG4,c

where:

  • ΔEIRPc = EIRPc(actual) − EIRPc(reference)
  • γG4,c = G4 step granularity for channel c
  • round() is rounding to the nearest integer
Quantization residual. The G4 correction cannot perfectly cancel the EIRP deviation because it is quantized to the G4 granularity. The residual is bounded by γG4,c / 2.

8.3 Phase 2: G4 to Final Targets

After all non-G4 gains reach their targets, G4 values will generally not be at their targets because of the accumulated compensation adjustments. Phase 2 uses the standard iteration loop (generate, filter, score, apply) restricted to G4 candidates only:

  1. Generate candidates from all stage types but keep only those with stageType === 'G4'.
  2. Filter by power thresholds (with deadlock relaxation).
  3. Filter by EIRP limits (if configured).
  4. Score and select the best G4 move.
  5. Apply and record until convergence.
EIRP deviation in Phase 2. During Phase 2, the G4 compensation cushion is removed as G4 moves toward its final target. This causes EIRP to deviate from initial values. If EIRP limits are configured, the algorithm obeys them, potentially requiring more (smaller) steps.

9. Deadlock Resolution

9.1 Cause

A deadlock occurs when no candidate move passes the power threshold check. This typically happens when the gain state is near a threshold boundary and every possible move in every direction would push some stage's output power outside its limits.

9.2 Resolution Mechanism

When all candidates fail the standard power threshold check, the algorithm retries with relaxed thresholds:

Plower'(k) = Plower(k) − 1.0    [dBm] Pupper'(k) = Pupper(k) + 1.0    [dBm] 
function filterRelaxedPower(candidates, gainValues, channels, gainStages):
    return candidates.filter(move =>
        tempGains = copy(gainValues)
        for each step in move.steps:
            tempGains[step.gainStageKey] += step.delta

        powerLevels = computeAllPowerLevels(channels, tempGains)

        for each (key, power) in powerLevels:
            stage = gainStages.get(key)
            if power > stage.upperThreshold + 1.0: reject
            if power < stage.lowerThreshold - 1.0: reject

        accept
    )

If relaxation produces feasible candidates, the iteration proceeds normally (the best move is selected from the relaxed set). The violations counter is incremented by 1.

9.3 Permanent Stuck

If even the relaxed check produces no feasible candidates, the algorithm terminates. The result includes all steps executed so far, with converged = false. The caller can inspect the partial result to understand where the transition stalled.

9.4 Interaction with EIRP Limits

Deadlock resolution applies only to power threshold checks. EIRP limit infeasibility is handled separately (see Section 6.4) and results in an error return, not a relaxation.

10. System Temperature Model

10.1 Friis Cascade Formula

The system noise temperature is computed at every step for visualization purposes. It uses the classical Friis formula applied to the receive chain (G1 → G2 → G3).

For channel c with receive chain stages indexed 1, 2, 3:

Tsys,c = Tant,c + ∑n=13 Tn / ∏m=1n−1 Gmlin

Expanding:

Tsys,c = Tant,c + TG1 + TG2 / G1lin + TG3 / (G1lin · G2lin)

10.2 Noise Temperature Conversion

Each stage's equivalent noise temperature is derived from its noise figure:

Tn = T0 · (Fn − 1) = 290 · (10NFn/10 − 1)    [K]

10.3 Computation Procedure

function computeChannelSystemTemp(channel, gainValues, gainStages):
    rxStageKeys = channel gain chain filtered to G1, G2, G3 only
    tSys = channel.antennaNoiseTemp
    cumulativeGainLinear = 1.0

    for i = 0 to rxStageKeys.length - 1:
        key = rxStageKeys[i]
        stage = gainStages.get(key)

        stageTemp = 290 * (10^(stage.noiseFigure / 10) - 1)    // T_n
        stageGainLinear = 10^(gainValues[key] / 10)             // G_n^lin

        tSys += stageTemp / cumulativeGainLinear
        cumulativeGainLinear *= stageGainLinear

    return tSys
Transmit side excluded. Only G1, G2, G3 contribute to the receive-side noise temperature. G4–G7 are on the transmit path and do not affect the receiver noise floor.

11. Convergence Analysis

11.1 Progress Guarantee

At each iteration, the selected move has score < 0, meaning it makes nonzero progress toward the target. The total gain delta across all stages is:

Δtotal = ∑k ∈ K |gk(T) − gk(0)| / γk    [steps]

Each iteration reduces this by at least 1 step (the minimum possible progress is a single-step move on one gain stage instance). Therefore, in the absence of deadlocks and EIRP infeasibility:

iterations ≤ Δtotal

11.2 Upper Bound

The worst case occurs when each iteration moves exactly one gain stage instance by one granularity step. With N total remaining steps across all gain stage instances, convergence is guaranteed in at most N iterations.

11.3 Typical Performance

Multi-step moves (multiplier > 1) and digital stage batching (all instances of a type moving together) dramatically reduce the iteration count. In practice, the algorithm converges in far fewer iterations than the worst-case bound.

11.4 Non-Convergence Cases

The algorithm may fail to converge in two scenarios:

  1. Permanent deadlock: All candidates are power-infeasible even with 1 dB relaxation. This indicates the gain configuration is physically unreachable from the current state without violating hardware limits.
  2. EIRP infeasibility: The EIRP limits are too tight for any available move. The algorithm returns an error with the best achievable deviation to guide the operator.

11.5 G4-Compensated Convergence

Phase 1 converges by the same argument as the standard algorithm, since each iteration makes progress on at least one non-G4 gain. Phase 2 converges because it reduces the total remaining G4 delta on each iteration. The total step count is bounded by:

steps ≤ 2 · Δnon-G4 + ΔG4-final

The factor of 2 accounts for the paired primary + G4 correction steps in Phase 1.

12. Correctness Justification

12.1 Invariant: Constraint Satisfaction

At every intermediate state recorded in the transition plan:

  • All power levels are within thresholds (possibly relaxed by at most 1 dB during deadlock resolution, tracked by the thresholdViolations counter).
  • All channel EIRP deviations are within configured limits (when EIRP limits are set).

This is maintained because every move is checked before application. No move is applied without passing the feasibility filters.

12.2 Invariant: Monotonic Progress

The scoring function ensures that each selected move has score < 0 (strictly negative), meaning nonzero gain delta is applied. Combined with the direction constraint (each gain stage moves only toward its target), the remaining delta decreases monotonically.

12.3 Invariant: Granularity Compliance

Every atomic step applies a delta that is an integer multiple of the stage's granularity. This is enforced by the candidate generator, which constructs deltas as direction * n * gran.

12.4 Termination

Monotonic progress on a finite quantity (total remaining steps) guarantees termination. The maxIterations parameter provides an additional safety bound (default: 5000).

12.5 G4-Compensated Correctness

In the G4-compensated strategy:

  • Phase 1 drives all non-G4 gains to their targets. G4 values may diverge from their targets, but this is intentional (they compensate).
  • Phase 2 drives G4 values to their targets using the standard loop. At termination, all 7 stage types are at their targets.
  • Both phases apply the same feasibility checks, maintaining the constraint satisfaction invariant throughout.

13. Computational Complexity

13.1 Per-Iteration Cost

Let C = number of channels, S = number of gain stage instances, M = max remaining steps, T = number of stage types with delta.

OperationComplexityNotes
Candidate generationO(T · M + A · M)A = analog antenna instances
Power feasibility check (per candidate)O(C · S)One EIRP/power cascade per channel
EIRP limit check (per candidate)O(C)One comparison per channel
Scoring (per candidate)O(|steps|)Sum of absolute deltas
Sorting scored candidatesO(|cands| · log|cands|)Standard comparison sort
State recordingO(C · S)EIRP + power + T_sys computation

The dominant cost is the power feasibility check, applied to every candidate. Per iteration:

O(|candidates| · C · S)

13.2 Total Complexity

With I iterations total:

O(I · |candidates| · C · S)

In typical payloads (C ≤ 100, S ≤ 700, M ≤ 20, I ≤ 200), the total computation takes under 1 second.

13.3 G4-Compensated Overhead

The G4-compensated strategy doubles the per-iteration cost due to:

  • EIRP limit checking both intermediate states (two EIRP evaluations per candidate instead of one).
  • G4 correction computation (one additional EIRP evaluation per selected move).
  • Recording two steps per iteration instead of one.

The overhead is constant-factor, not asymptotic.

14.2 Coupling Map

The coupling map is a Map<string, string[]> mapping each gain stage key to the list of channel IDs that flow through it. It is built once during initialization and consulted during G4 correction computation.

14.3 State Representation

The system state is maintained as a flat Record<string, number> mapping gain stage keys to their current dB values. This allows O(1) lookup and update per gain stage. A deep copy is made for each candidate evaluation to avoid mutation.

16. Limitations & Future Work

16.1 Current Limitations

  • Greedy local optimality. CCGS uses a greedy strategy that maximizes progress at each step. It does not guarantee the globally optimal transition sequence (minimizing some aggregate cost function over the entire path). A globally optimal solution would require exploring the full state space, which is combinatorially intractable for realistic payload sizes.
  • Fixed deadlock relaxation. The 1 dB threshold relaxation is fixed. An adaptive relaxation strategy (e.g., binary search for the minimum relaxation needed) could reduce unnecessary threshold violations.
  • G4-only compensation. The compensation strategy is limited to G4. In principle, any per-channel stage (G3, G5) could serve as a compensator, or multiple stages could be used in combination.
  • Single-type moves. Each iteration changes only one gain stage type. Allowing coordinated multi-type moves could reduce the total transition time but would exponentially increase the candidate space.
  • No temporal constraints. The algorithm does not model timing—it assumes each step takes equal time. Real hardware may have stage-dependent settling times.

16.2 Future Work

  • Look-ahead scoring. Evaluating candidates based on their 2–3 step lookahead impact rather than single-step progress.
  • Adaptive EIRP weighting. Reintroducing asymmetric cost functions that penalize negative EIRP deviations (service degradation) more heavily than positive deviations (potential interference), combined with the hard EIRP limits.
  • Parallel transitions. Modeling hardware that can adjust multiple stage types simultaneously, reducing total transition time.
  • Constraint learning. Using the results of past transitions to inform future optimization (warm-starting from similar configurations).
  • Multi-compensator strategies. Extending the G4-compensated approach to use G3 or G5 as secondary compensators when G4 granularity is too coarse.
