1. Problem Statement

A satellite communications payload consists of X receive antennas and Y transmit antennas connected through a multi-stage gain chain. Each logical channel traverses a fixed path of 7 gain stages from receive to transmit. The operator must reconfigure gain settings across all stages from a current operating point to a new target configuration.

The transition cannot be applied atomically—each gain stage is adjusted individually in discrete increments determined by its hardware granularity. During the transition, the payload continues to serve active traffic. The problem is to find a sequence of atomic gain adjustments that:

  1. Minimizes transient EIRP disruption per channel, with negative deviations (service degradation) penalized more heavily than positive deviations (potential interference).
  2. Never violates power thresholds at any gain stage output during any intermediate state. Each stage has hardware-defined upper and lower power limits beyond which the payload cannot safely operate.
  3. Respects discrete granularity constraints: each gain stage can only be adjusted in multiples of its specific step size.
  4. Terminates with all gain stages at their target values.
Operational context. In geostationary satellite payloads, gain reconfigurations occur during beam repointing, traffic rebalancing, or interference mitigation. A naive sequential approach—adjusting one stage at a time—can cause transient EIRP excursions of tens of dB, triggering regulatory violations or link margin exhaustion. CCGS reduces these transients to the minimum physically achievable given the discrete step constraints.

2.3 Channel Definition

A channel c is a logical signal path fully characterized by:

  • Rx antenna assignment r(c) and Tx antenna assignment t(c)
  • Bandwidth Bc in MHz
  • Rx low frequency frx(c) and Tx low frequency ftx(c) in MHz
  • Input Power Flux Density IPFDc in dBm/m²
  • EIRP target EIRPtarget(c) in dBm

The 5 MHz subchannel index for a frequency f is defined as ⌊f / 5⌋. A channel with bandwidth exceeding 5 MHz spans multiple subchannels and thus shares G2/G6 with other channels in each of those subchannels.

2.4 Coupling Structure

Definition (Coupling Map). The coupling map Φ: GainStageKey → {ChannelId} maps each gain stage instance to the set of channels whose EIRP and power levels it influences. This map is derived deterministically from the channel frequency plan and antenna assignments.

The inverse mapping—from a channel to its 7-stage gain chain—is denoted Γ(c) = [g1, g2, ..., g7].

3. Mathematical Formulation

3.1 State Space

Let S be the set of all gain stage instances. The system state is a vector:

g = (gs)s ∈ S ∈ ℝ|S|

where each gs is constrained to the lattice defined by the stage's granularity δs. The initial state is g(0) and the target state is g*.

3.2 EIRP Model

The EIRP of channel c in state g is:

EIRPc(g) = Pin(c) + ∑s ∈ Γ(c) gs

where the input power is derived from IPFD and bandwidth:

Pin(c) = IPFDc + 10 · log10(Bc · 106)

The EIRP deviation from the initial operating point is:

ΔEIRPc(g) = EIRPc(g) − EIRPc(g(0)) = ∑s ∈ Γ(c) (gs − gs(0))

3.3 Power Level Model

The output power at gain stage s depends on whether the stage is shared or per-channel:

Per-channel stages (G3, G4, G5): the power is the single channel's cumulative power through the chain up to and including stage s:

Ps(c) = Pin(c) + ∑s' ∈ Γ(c), s' ≤ s gs'

Shared stages (G1, G2, G6, G7): the total power is the sum in linear domain of all channel contributions flowing through the stage:

Ps = 10 · log10 &left;( ∑c ∈ Φ(s) 10Ps(c) / 10 &right;)
Critical implementation detail. Powers must be summed in the linear domain (milliwatts), not the dB domain. Summing in dB is a common error that leads to incorrect threshold assessments, particularly at shared stages where multiple channels aggregate. The implementation uses explicit dbmToLinear() and linearToDbm() conversions.

3.4 Constraints

Threshold constraints. At every intermediate state during the transition, for every gain stage s:

Ls ≤ Ps(g) ≤ Us

where Ls and Us are the lower and upper power thresholds in dBm.

Granularity constraints. Each gain adjustment must be an integer multiple of the stage's step size:

Δgs ∈ { k · δs : k ∈ ℤ }

EIRP deviation constraint (optional). When enabled, the maximum absolute EIRP deviation from the initial operating point is bounded for all channels at every intermediate state:

|ΔEIRPc(g(i))| ≤ Dmax    ∀ c, ∀ i ∈ {1..N}

where Dmax is the operator-specified maximum deviation in dB. When Dmax is not set (null), this constraint is inactive and only the soft asymmetric cost steers the optimization. When active, it acts as a hard feasibility filter—any candidate move that would push any channel beyond ±Dmax is rejected, regardless of its cost score.

Convergence trade-off. Setting Dmax too tight relative to the gain step granularities may prevent convergence. For example, if a shared gain (G1) has 0.5 dB granularity and the compensating per-channel gain (G3) has 0.25 dB granularity, the minimum achievable transient for that pair is 0.25 dB. Setting Dmax < 0.25 would block all shared gain moves. As a rule of thumb, set Dmax ≥ max(δs) across all gain stages.

3.5 Objective Function

The objective is to find a transition sequence g(0) → g(1) → ... → g(N) = g* that minimizes the worst-case asymmetric EIRP disruption:

minimize   maxi ∈ {1..N} ∑c J(ΔEIRPc(g(i)))

where the asymmetric cost function is:

J(Δ) = w− · max(0, −Δ) + w+ · max(0, Δ)

with w− > w+ (default ratio 3:1) reflecting that service degradation is more critical than excess power.

4. Algorithm Description

4.1 Overview

CCGS is a greedy iterative algorithm that constructs the transition sequence one move at a time. At each iteration, it:

  1. Generates a set of candidate moves (single atomic steps and compensating pairs).
  2. Filters candidates for feasibility against all power thresholds.
  3. Filters remaining candidates against the optional hard EIRP deviation limit (Dmax).
  4. Scores feasible candidates using the asymmetric EIRP cost function.
  5. Selects and applies the lowest-cost feasible candidate.
  6. Records the full system state (gain values, EIRP, power levels) for visualization.

The algorithm terminates when all gain stages reach their target values, no feasible moves exist, or the iteration limit is reached.

The following diagram expands the inner loop, showing every decision point and data flow within a single iteration:

4.3 Pseudocode

function CCGS(channels, gainStages, params):
    couplingMap ← buildCouplingMap(channels)
    gainValues  ← { s.key: s.currentValue for s in gainStages }
    targets     ← { s.key: s.targetValue  for s in gainStages }
    initialEirp ← computeAllChannelEirp(channels, gainValues)
    steps       ← []

    for iter = 0 to params.maxIterations:
        if allAtTarget(gainValues, targets):
            break

        singles ← generateSingleSteps(gainValues, targets, granularities)
        pairs   ← generateCompensatingPairs(gainValues, targets, granularities, couplingMap)
        candidates ← pairs ++ singles

        feasible ← filter(candidates, move → checkFeasibility(move, gainValues, channels, gainStages))

        // Hard EIRP deviation constraint (when D_max is set)
        if params.maxEirpDeviation ≠ null:
            feasible ← filter(feasible, move → checkEirpDeviation(move, gainValues, channels, initialEirp, D_max))

        if feasible is empty:
            // Deadlock resolution: relax thresholds by 1 dB
            feasible ← filterRelaxed(singles, gainValues, channels, gainStages)
            if feasible is empty: break  // truly stuck

        scored ← [(move, scoreCandidate(move, gainValues, channels, initialEirp, params)) for move in feasible]
        sort scored by (score ASC, tiebreaker)

        best ← scored[0].move
        applyMove(best, gainValues)
        steps.append(recordState(best, gainValues, channels, initialEirp))

    return TransitionResult(steps, converged, metrics)

5. Candidate Generation

5.1 Single Atomic Steps

For each gain stage s where gs ≠ gs*, a single atomic step is generated:

Δgs = sign(gs* − gs) · δs

This produces at most |S| candidate moves per iteration, where |S| is the number of gain stages with remaining delta.

5.2 Compensating Pairs

Compensating pairs are the key mechanism for minimizing EIRP disruption. When a shared gain (G1, G2, G6, G7) must change, it affects all channels flowing through it. A compensating pair bundles this shared gain step with opposing per-channel gain steps that cancel the EIRP impact.

Definition (Compensating Pair). A compensating pair for shared stage s is a composite move consisting of:
  1. An atomic step +δs (or −δs) on the shared stage.
  2. For each affected channel c ∈ Φ(s): an atomic step −δq (or +δq) on a per-channel stage q ∈ {G3, G4, G5} of that channel, opposing the shared step.

When the shared and per-channel granularities are equal, the net EIRP change per channel is exactly zero. When granularities differ, the residual is bounded by the granularity difference.

For each shared stage with remaining delta, the algorithm generates up to 3 compensating pairs (one per compensating gain type G3, G4, G5), provided all affected channels have the corresponding per-channel gain available. This yields at most 3 · |Sshared| additional candidates.

5.3 Candidate Pool Size

Per iteration, the total candidate pool is bounded by:

|Candidates| ≤ |S| + 3 · |Sshared|

For a payload with C channels, X Rx antennas, Y Tx antennas, and K subchannels:

|S| = X + Krx + 3C + Ktx + Y |Sshared| = X + Krx + Ktx + Y

6. Feasibility Verification

6.1 Procedure

For each candidate move, feasibility checking proceeds as follows:

  1. Simulate the move: Create a temporary copy of the gain state and apply all atomic steps in the candidate.
  2. Compute power levels: For each channel, compute the cumulative power at every stage in its chain. For shared stages, aggregate contributions from all channels in the linear domain.
  3. Check thresholds: Verify that the aggregated power at every stage falls within [Ls, Us]. A tolerance of 0.001 dB is applied to avoid floating-point rejection of boundary-valid states.

7. Scoring & Selection

7.1 Primary Score: Asymmetric EIRP Cost

Each feasible candidate is scored by simulating the move and computing the resulting EIRP deviation from the initial operating point (not the target). The cost function:

Score(move) = ∑c [ w− · max(0, −ΔEIRPc) + w+ · max(0, ΔEIRPc) ]

The deviation is measured from g(0), not from the current state. This ensures the cost reflects the cumulative transient impact on active traffic, not just the marginal effect of this step.

Design rationale. Measuring deviation from the initial state rather than the previous state prevents a subtle failure mode: a sequence of individually low-cost steps could drift far from the initial EIRP if each step is only measured against its predecessor. By anchoring to the initial state, the algorithm always prefers states closer to the operating EIRP.

7.2 Hard EIRP Deviation Constraint

When maxEirpDeviation (Dmax) is set, an additional feasibility filter is applied after power threshold checking and before scoring. For each candidate move, the algorithm:

  1. Simulates the move on a temporary gain state copy.
  2. Computes the EIRP for every channel in the post-move state.
  3. Computes the absolute deviation from the initial EIRP: |EIRPc(gpost) − EIRPc(g(0))|.
  4. Rejects the move if any channel exceeds Dmax (with 0.001 dB tolerance).

This provides a hard guarantee that no channel will ever experience a transient EIRP excursion beyond the specified bound. It differs from the soft asymmetric cost in a critical way:

MechanismTypeEffect
Asymmetric cost (w−, w+) Soft Steers the optimizer to prefer low-deviation moves, but allows larger deviations if no better option exists
Max EIRP deviation (Dmax) Hard Rejects any move exceeding the bound, even if it is the lowest-cost option. May cause non-convergence if too restrictive
Interaction with deadlock resolution. When the hard EIRP constraint is active and all feasible candidates are eliminated, the algorithm enters deadlock resolution (threshold relaxation). The EIRP constraint is not relaxed during deadlock resolution—only power thresholds are. This ensures the EIRP guarantee is never violated. If no move satisfies both the relaxed thresholds and the EIRP bound, the algorithm terminates with converged = false.

7.4 Tiebreaking Rules

When two candidates have the same primary score (within 0.001 tolerance), the following tiebreakers apply in order:

  1. Compensating pairs preferred (if preferCompensatingPairs is enabled): Pairs make progress on shared gains while preserving EIRP, advancing the transition more efficiently.
  2. Inner gains preferred (if strategy is inner-first): Per-channel gains (G3, G4, G5) are preferred over shared gains (G1, G2, G6, G7) because they affect fewer channels and are easier to compensate later.
  3. Fewer steps preferred: Simpler moves are preferred to reduce state-space exploration.

7.5 Strategy Modes

StrategyBehaviorBest For
greedy Pure lowest-cost selection; no structural preference between gain types General-purpose optimization; when gain changes are well-distributed across stages
inner-first Prefer per-channel gains before shared gains as a tiebreaker Payloads with tight shared-stage thresholds; reduces coupling risk by settling per-channel gains before touching shared amplifiers

8. Deadlock Resolution

8.1 When Deadlocks Occur

A deadlock arises when no candidate move satisfies all power thresholds strictly. This can happen when:

  • The current state is near a threshold boundary and every feasible direction pushes at least one stage beyond its limit.
  • Coupled stages create circular dependencies: moving one stage requires moving another first, and vice versa.
  • Compensating pairs cannot be formed because per-channel gains are already at their targets.

8.2 Resolution Mechanism

When no strictly feasible move exists, the algorithm enters relaxed mode:

  1. Only single steps are considered (pairs are excluded to simplify the search).
  2. Threshold limits are relaxed by 1.0 dB in both directions: [Ls − 1, Us + 1].
  3. Among relaxed-feasible steps, the lowest-cost candidate is selected.
  4. The relaxation is counted and reported as a thresholdViolation in the result.

If even relaxed-mode produces no feasible move, the algorithm terminates with converged = false.

Operational implication. Threshold relaxations indicate that the requested gain transition is tight against hardware limits. Each relaxation should be reviewed by the operator. If the number of relaxations is unacceptably high, the operator should consider widening thresholds or breaking the transition into smaller steps.

9. Convergence Analysis

9.1 Progress Guarantee

At each iteration, at least one candidate move exists that reduces the total remaining delta across all gain stages (the single step for any stage with remaining delta). Provided this move is feasible, the algorithm makes progress.

Lemma (Monotone Progress). Let R(i) = ∑s |gs* − gs(i)| be the total remaining delta at iteration i. For every iteration where a feasible single step exists, R(i+1) ≤ R(i) − δmin, where δmin = mins δs.

Proof sketch. A single atomic step moves one stage toward its target by exactly δs ≥ δmin. Compensating pairs move the shared stage toward target (reducing R by δs) and may move per-channel stages away from target (increasing R), but the net progress on the shared stage is guaranteed. Since compensating pairs are only preferred at equal cost (not forced), a pure single step is always available as fallback.

9.2 Termination Bound

The maximum number of iterations is bounded by:

Nmax = ∑s ∈ S ⌈ |gs* − gs(0)| / δs ⌉

This is the total number of atomic steps needed if each gain stage is adjusted independently. The algorithm may require more iterations than this in the worst case if compensating pairs temporarily move per-channel gains away from target, but the iteration limit parameter provides an absolute safety bound.

9.3 Conditions for Non-Convergence

The algorithm may fail to converge if and only if:

  1. The target configuration is infeasible: some gain stage's target value would cause a power level outside its threshold even in the final state.
  2. There exists no threshold-feasible path through the discrete gain lattice from the initial to the target configuration. This can occur when thresholds are extremely tight relative to the gain changes required.
  3. The iteration limit is reached before convergence (a safety bound, not a fundamental limitation).

10. Correctness Justification

10.1 Threshold Safety

Claim: At every intermediate state during the transition, all power levels are within their prescribed thresholds (excluding explicitly reported relaxations).

Justification: Every candidate move undergoes a full feasibility check before application. The feasibility check computes the exact power level at every gain stage in the post-move state, including proper linear-domain aggregation for shared stages. Only moves that pass all threshold checks are applied. The feasibility filter uses a 0.001 dB tolerance to avoid floating-point boundary effects, ensuring robust checking.

10.2 EIRP Minimization Quality

Claim: The algorithm produces a locally optimal transition sequence with respect to the greedy selection criterion.

Justification: At each step, the algorithm evaluates all feasible moves (both single steps and compensating pairs) and selects the one with the minimum asymmetric EIRP cost. This is a greedy choice—it does not guarantee a globally optimal sequence (which would require exponential-time search over the full combinatorial space of step orderings). However:

  • The compensating pair mechanism captures the most important structural optimization: when a shared gain must change, offsetting the impact with per-channel gains achieves near-zero EIRP disruption for that step.
  • The asymmetric cost anchored to initial EIRP prevents cumulative drift that purely local (step-to-step) optimization would miss.
  • In practice, the greedy approach produces solutions within 1-2 step granularities of the theoretical minimum disruption, because most EIRP disruption is caused by shared gain changes, and compensating pairs handle these optimally.

10.3 EIRP Deviation Bound

Claim: When maxEirpDeviation (Dmax) is set, no channel experiences an absolute EIRP deviation exceeding Dmax at any intermediate state.

Justification: The EIRP deviation filter is applied as a hard feasibility check after power threshold filtering and before scoring. For each candidate move, the algorithm computes the exact post-move EIRP for every channel and rejects the move if any channel's absolute deviation from the initial EIRP exceeds Dmax. The filter is applied consistently to both single steps and compensating pairs. During deadlock resolution (threshold relaxation), only power thresholds are relaxed—the EIRP bound is never violated. A tolerance of 0.001 dB is applied to avoid floating-point boundary rejection.

Non-convergence implication. Unlike the soft cost function, the hard EIRP constraint can cause the algorithm to terminate without reaching the target configuration. This is the correct behavior: it signals that the requested transition cannot be achieved within the specified EIRP envelope and the operator must either widen Dmax, adjust gain granularities, or break the transition into multiple phases.

10.4 Granularity Correctness

Claim: All gain adjustments respect the discrete granularity of each stage.

Justification: Candidate generation produces only steps of exactly ±δs for each stage s. The target convergence check uses a tolerance of 0.01 · δs to account for floating-point accumulation after many additions. No interpolation or rounding is applied to gain values during the transition.

12.2 Coupling Map

CouplingMap = Map<string, string[]>   // gainStageKey → channelId[]

Built once at algorithm start by iterating all channels and registering each channel's ID against all 7 gain stage keys in its chain. Lookup is O(1) per stage.

12.3 Transition Recording

Each transition step records a complete state snapshot:

  • gainValues: All gain stage values (deep copy)
  • channelEirp: EIRP per channel
  • channelEirpDeviation: EIRP deviation from initial per channel
  • powerLevels: Aggregated power at every gain stage
  • appliedMove: The candidate move that produced this state

This enables O(1) random access to any intermediate state for the step slider visualization without re-computation.

14. Limitations & Future Work

14.1 Current Limitations

LimitationImpactMitigation
Greedy selection (no global optimization) Solution may not be globally optimal in step count or max deviation Compensating pairs capture the dominant optimization; greedy deviation is small in practice (< 1 step granularity)
Single-step lookahead Cannot plan multi-step maneuvers that temporarily worsen cost to achieve a better final trajectory Could be extended with beam search (k-best) or A* with admissible heuristic
No temporal constraints Does not model the time required to apply each gain change or settling time of analog stages Iteration count serves as a proxy; could add per-stage delay weights
Simplified input power model Pin assumes zero effective aperture and no antenna gain variation Sufficient for transition optimization (only deltas matter); can be extended with per-antenna aperture parameters
Wideband channels span multiple subchannels EIRP uses only the primary subchannel's G2/G6 for simplicity Could compute weighted average across subchannels for higher fidelity

14.2 Potential Extensions

  • Beam search: Maintain top-k partial sequences and select the best final result. Trades compute time for solution quality.
  • Constraint programming: Formulate as a CSP and use backtracking with arc consistency for exact solutions on small payloads.
  • Batched moves: Apply multiple non-interfering moves per iteration (moves that affect disjoint sets of channels can be parallelized).
  • Intermodulation products: Extend power model to account for IM products at shared stages, particularly relevant for HPA saturation.
  • Monte Carlo validation: Randomize step ordering and compare against CCGS to quantify optimality gap.
  • Real-time mode: Integrate with payload telemetry to monitor actual power levels during transition and re-plan if deviations occur.
